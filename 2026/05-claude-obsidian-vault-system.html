SECTION 1 ／ 為什麼是 Obsidian

實測：兩個月
累積到了什麼、還沒做到什麼

Obsidian 的學習成本確實比 Notion 高。下面用兩個月的實測數據檢視：這個成本是否值得、目前實際做到什麼、還沒做到什麼。

知識工作者面對的兩個挑戰

編輯工作要解決兩個完全不同性質的問題：個人累積（寫過的判斷怎麼讓未來的稿件繼承）與團隊協作（稿件流程、審稿、跨人交接）。單一工具很難同時解決，Obsidian 解第一個、Notion 解第二個。我同時用，不是替代關係。

Obsidian 與 Notion 在我工作流的分工

功能需求 用哪個 為什麼這樣分
個人素材吸收（inbox） Obsidian Web Clipper（Obsidian 的網頁素材抓取插件）一鍵存進 vault、純文字易批次處理、AI 可以全 vault 餵
知識累積與連結（research） Obsidian [[]] 雙向連結、本地檔案、用搜尋指令就能找到、不受廠商綁定
個人專案文件（projects） Obsidian 沒有審稿流程，給自己看的工作筆記
每日工作日誌 Obsidian 純文字、寫起來零摩擦，未來可被 grep 或 AI 直接處理
跨人稿件協作 Notion 審稿狀態、屬性欄位、view filter、多人同時編輯，這是 Obsidian 做不到的
稿件主檔（single source of truth） Notion 發布前的 review、發布後的歸檔，整個編輯部共看一個資料庫
主管／同事跨人查找 Notion 權限管理、查詢介面、不需技術門檻
結論：兩個工具各司其職。Obsidian 是個人知識庫、Notion 是編輯部共用工作台。重點不是選哪一個，而是讓兩邊都做它最擅長的事。

Obsidian 對知識工作者的三個累積機制

Obsidian 對知識工作者的差異點不是「比 Notion 好」，而是有三個 Notion 沒有的累積機制：

軸線 01

連結網：相關性自己浮現

不靠資料夾分類，靠 [[]] 雙向連結。寫新筆記時，輸入相關詞會自動帶出舊筆記，幫你看見早就寫過的內容。

軸線 02

本地主權：你寫的真的是你的

Plain text + Markdown，10 年後不管哪個工具還在，這些檔案都打得開。Notion 漲價、倒站、改規則都不影響你，這對「累積資產」性質的工作極重要。

軸線 03

AI 整合：vault 越完整，AI 越能對齊你的工作脈絡

我的 Claude 認得我寫過什麼、人物譯名、產業判斷。沒有把個人脈絡系統化餵給 AI 的人，得到的回答往往偏通用、跟自己工作脈絡對不上。

實測限制：這套系統目前還沒做到什麼

誠實揭露 4 個目前還沒驗證的點：

  • 稿件層次的累積還沒發生：vault 內部 9 個檔案引用 Harness，但 archive 16 篇發布稿件中 0 篇引用任何 vault 累積的 research。也就是說「research 累積」目前只發生在 vault 內部，還沒轉化成「稿件大量取用」。兩個月還看不到這個轉化的證據。
  • 維護機制剛建立、還沒被驗證：50 條 memory 警戒線、升級規則、lint-vault 健康檢查擴充，這些自動 audit 機制是最近才寫進系統的，還沒驗證 3 個月以上能不能真的避免膨脹。設計上有出口不等於實證上會自動運作。
  • 對非技術同事門檻高：Obsidian 安裝、[[]] 語法、Claude Code 操作，對非技術背景的人不友善。後面 CTA 的「倒著建」步驟其實只對能跑 Claude Code 的人有用。
  • 「複利」還沒用稿件產出指標驗證：真正的複利應該是寫稿時間變短、稿件品質變好。這兩個指標目前沒有數據可以證明。本文展示的是「vault 內檔案彼此引用」的密度指標，可以視為未來複利的潛力，不是已實現的複利。

誰不適合用 Obsidian

除了上面的實測限制，這套系統對下面幾類人也不會產生累積效果：

  • 純流水帳作者：每天記了就記了，未來不會回頭查找。累積的前提是「會被再次取用」。
  • 工作不需要累積知識的崗位：執行型任務、SOP 固定的工作。
  • 沒有寫作累積習慣的人：你必須先有「想寫下來」的動機，工具才幫得上忙。
  • 沒辦法忍受 2 到 4 週適應期的人：Obsidian 確實有學習曲線，前幾週你會覺得「我為什麼花時間整理這個」，直到第一次「啊那篇我有寫過」浮現時，你才會 get 它的價值。
SECTION 2 ／ 核心比喻

失憶的萬能寫手

在進入分工架構之前，先用一個比喻把整套系統的角色定位釐清。

Claude 是一個能力極強、但每天上班會失憶的寫手。你給他一支筆、放他一個人在房間裡，他什麼都做得到，前提是你得告訴他「你是誰、我是誰、我們在做什麼」。
我的 Obsidian vault + Memory + Skills 架構，就是這個「告訴他」的整套機制。
整套系統的核心邏輯

所以接下來要解釋的，是「告訴他」這個動作怎麼被拆成四層（從通用到具體）：

  • 通用原則（CLAUDE.md + Memory）：每次對話自動載入的工作守則與個人脈絡
  • 任務 SOP（Skills）：特定稿件類型的標準做法，被觸發時才讀取
  • 知識素材（Vault）：產業背景與過往判斷的檔案室，要查才查
  • 對外協作（Notion）：稿件審稿與跨人協作的目的地
SECTION 3 ／ 分工架構

分工架構：從通用到具體

整套系統不是並列的四層，是一個逐層補充的協作結構。上層提供「通用原則／個人脈絡」，下層補上「任務 SOP／實際資料」，四層互補協作，不是上層管下層。Skill 啟動時會主動讀對應的風格規範檔（譬如 bwt-style-guide）；CLAUDE.md 與 Memory 則在對話啟動時自動載入，所有層共用同一份。這是設計時的同步紀律。

LAYER 0

通用原則：CLAUDE.md ＋ Memory

每次對話啟動就載入。CLAUDE.md 是全域守則（怎麼工作），Memory 是跨對話的個人脈絡（譯名、偏好、個案教訓）。下面所有任務都會吸收這層的內容。

通用層
ALWAYS LOADED
↓ 補上 SOP（任務專屬流程接續通用原則）
LAYER 1

任務 SOP：Skills

被觸發時才讀取，把「這類稿件怎麼寫」「這類配件怎麼做」的具體流程補上。Skill 內主動引用上層的風格規範與個人偏好，不靠即時運作時強制覆寫。

純資料層，提供事實、背景、過往判斷。Skill 引用素材時的依據，不下指令也不影響規則。

LAYER 3

引擎層。提供通用智慧、語言處理、推理能力。上面三層都是「告訴它怎麼用這份能力」的工程。

各層的協作關係：這四層不是「上層管下層」的權限階梯，是「逐層補充」的同步紀律。當我修改 Skill 或 CLAUDE.md 時，Claude 會主動 Grep 下游引用確認一致；lint-vault 偵測到同主題 memory ≥ 3 條時，會建議升級到 Skill 規範檔。整個系統是「從個案累積、升上去」的演化，不是即時運作時的權限覆寫。

階梯之外還有一層：對外協作

上面四層是「對 Claude 的指令與資料」，但稿件最後要交回編輯部審稿。這就需要一個「跨人協作」的目的地，對我來說是 Notion。

EXTERNAL
對外協作

Notion：編輯部共用的辦公桌

Notion 不在前面的階梯結構裡，因為它不是 Claude 讀取的「層」，而是 Claude 寫回的目的地。它解決的是 Obsidian 解不了的問題：

  • 稿件主檔：發布前審稿、發布後歸檔，整個編輯部共看同一份
  • 權限／審稿狀態／屬性欄位：跨人協作必備，Obsidian 做不到
  • 非技術同事也能用：主管、業務、其他部門都查得到

所以整套系統的完整形貌是：對內逐層補充（Claude × CLAUDE.md × Memory × Skills × Vault）+ 對外協作目的地（Notion）。前者是個人累積、後者是團隊共用，兩者各司其職。

SECTION 4 ／ 實例追蹤

「跑稿件」會發生什麼：
一句指令背後的5 步流程

把抽象的分工變具體：當我跟 Claude 說「跑先泰今天的稿件」，這句話背後到底發生了什麼？

01

Claude 醒來，載入「人格」

自動讀取 CLAUDE.md（員工守則）+ MEMORY.md（個人交接本）。現在 Claude 知道我是《數位時代》編輯、產出存哪裡、付費牆讀不到就停、Mark Gurman 譯名是古爾曼。

→ 載入：CLAUDE.md 全域 + 專案 + MEMORY.md 索引（43 條目摘要）
02

從句子認出「跑稿件」+「先泰」

Claude 觸發 notion-orchestrator Skill，這時才去讀那個 skill 檔案。平常它不載入，現在才付這個 context 成本。

→ 觸發：~/.claude/agents/notion-orchestrator.md（按需召喚）
03

orchestrator 派發 sub-skill

掃描 Notion 資料庫找先泰名下的「未開始」稿件，依 Notion「稿件類型」欄位派發：即時新聞 → news-daily、深度分析 → deep-analysis、教學文 → tutorial-article、潤稿 → draft-polish 等。每個 sub-skill 啟動時會主動 Read bwt-style-guide（風格規範）。

→ 再載入：對應 sub-skill + bwt-style-guide.md + bwt-voice-reference.md
04

子代理依素材寫稿

子代理拿到 orchestrator 傳入的素材（從 Notion「來源」欄位讀來的 URL 或本地檔）開始寫稿。URL 來源會先比對 inbox/ 找 Web Clipper 抓好的本地檔；找不到才用 web_fetch。inbox/ 是 orchestrator 自動撈的範圍，research/ 屬獨立流程（要使用者明確指定才用）。

→ 按需讀取：Notion 條目 + ~/Claude Project/inbox/（已預先匹配的本地檔）
05

成品寫回 Notion，跑查核

稿件完成後寫回 Notion 對應頁面，自動跑 article-checker 做錯字、事實、貨幣換算查核。Notion 是稿件主檔，編輯部所有人在同一份。

→ 觸發：article-checker skill + Notion MCP 寫回
關鍵觀察：常駐人格（CLAUDE.md + Memory）在對話開頭載入一次，提供「人格」（規則 + 偏好）；後續每個具體動作大量依賴「按需召喚」（Skills + Vault）。這就是為什麼前者要嚴格管理上限、後者可以放心擴充。
SECTION 5 ／ 設計原則

為什麼這樣分？
一句話的成本邏輯

整套系統的設計可以濃縮成一句話：

每次對話開頭都載入，要花成本；按需召喚，不用就不花。
所以規則放上面、知識放下面。

延伸：四種放置原則

內容性質 放哪裡 原因
通用鐵則 CLAUDE.md 常駐，永遠執行
個人偏好／教訓 Memory 常駐，但要管理上限避免膨脹
任務 SOP Skill 按需，可以無限擴充
知識／歷史 Vault 按需查詢，零常駐成本

這也解釋了一個常見疑問：為什麼我會花時間清理 memory？因為它是「房租」，越多越貴。Skill 不用清，要多少有多少。

SECTION 6 ／ 常見誤解

同事最常問的 5 個問題

Q1.不能全部都放 memory 嗎？反正都會被記住。

Memory 每次對話開頭都要載入，會吃掉「對話記憶空間」（context window）。超過一定量會讓 Claude 反應變慢，甚至擠壓真正在做的任務空間。Skill 按需召喚，不用就不花成本。所以我設定了 50 條的容量警戒線。

Q2.Skill 跟 memory 差在哪？感覺都是「告訴 Claude 怎麼做」。

Memory 是事件性／偏好性（「我」是誰、「我」怎麼做事），Skill 是任務性／流程性（「這種稿件」怎麼寫）。換個人來用，memory 要重寫，但 Skill 可以共用，這就是為什麼編輯部共用 BN_next 共享庫（GitHub repo）同步 Skills，但 memory 是我個人的。

Q3.Vault 為什麼不直接放 memory？

Vault 是「給人讀」的，研究、歸檔、選題用。Memory 是「給 Claude 讀」的，跨對話人格。Vault 可以無限大、可以放圖片影片；memory 一定要精簡，因為要常駐。

Q4.我也想用，怎麼開始？

倒著建。先建 vault（inbox + research 兩個資料夾就夠開始），用一陣子後發現「每次都要重複告訴 Claude 同一件事」，那件事就升級到 CLAUDE.md 或 memory。當某類任務你做超過 3 次發現 SOP 穩定了，就寫成 Skill。不要一開始就建全套架構，會卡在工具設計而沒在做事。

Q5.這套系統不會越積越亂嗎？

會，所以要有清理機制。我的維護工具包含：lint-vault skill（每月一次的 vault + memory 健康檢查，掃孤兒筆記、過時內容、容量警戒）、每日收工流程（inbox 清理、draft 歸檔、Wiki Ingest 候選清單）、Memory 升級規則（同主題累積 ≥ 3 條就升 Skill 規範檔，避免 memory 無限膨脹；50 條 / 12,000 字元設容量警戒線）。「有自動健康檢查」是這套系統的差異化點。Notion 在跨人協作上強，但個人累積層級的健康狀況需要另外設計檢查流程；這套系統把定期體檢直接內建進來。

怎麼開始用

想試的話：
不要一開始就建全套

整套架構聽起來複雜，但你不需要一次到位。我的實測經驗是「倒著建」：從最小可行開始，遇到痛點才往上長。下面是兩個月走過的軌跡：

  1. 第 1 週：裝 Obsidian，只建 inbox/ 與 research/。用 Web Clipper 抓素材，看到值得記的就寫一篇 research。
  2. 第 2-4 週：累積到 10 篇 research 後，你會發現有些主題反覆出現，這時候才開始建 [[]] 連結。
  3. 第 2 個月：發現「每次都要跟 Claude 講同一件事」時，把那件事寫進 CLAUDE.md 或 memory。
  4. 第 3 個月以後：某類任務做超過 3 次發現 SOP 穩定了，寫成 Skill。團隊共用的 SOP 同步到 GitHub 共享庫。
  5. 第 4 個月起：每月跑一次 lint-vault skill，把孤兒筆記、過時內容、超過警戒線的 memory 列出來批次處理。維護是累積能持續發酵的前提。
